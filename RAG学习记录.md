# LangChain + RAG 技术要点笔记

## 一、LangChain 基础

### Q1: 什么是 LangChain？
LangChain 是一个专为构建大语言模型（LLMs）应用而设计的开发框架。

**官方定义**：  
LangChain is a framework for developing applications powered by large language models (LLMs).

**典型应用场景**：
- Chatbots 聊天机器人
- 基于 RAG 的问答系统
- 基于 SQL 数据库的智能查询
- 工具调用与 Agents 智能代理

> 官方用例参考：https://python.langchain.com/docs/use_cases/

## 二、RAG 核心概念

### Q2: 什么是 RAG（检索增强生成）？
RAG 是指对大型语言模型输出进行优化，使其在生成响应前能够引用训练数据之外的外部权威知识库。

**技术本质**：基于知识库检索的提示词增强技术。  
**工作流程**：根据用户输入在向量数据库中检索相关信息，将检索结果与原始提示词共同送入 LLM，提升回答质量。

### Q3: RAG 的主要优点
- **减少幻觉**：通过注入真实知识片段，有效抑制模型编造事实。
- **数据安全**：检索内容仅作为 Prompt 输入，用户不可见，降低数据泄露风险。
- **避免频繁微调**：知识库可动态更新，无需重新训练模型，节约成本。

### Q4: RAG 面临的主要挑战
- **检索准确性**：如何确保召回的内容与问题高度相关。
- **生成编排**：如何在有限的上下文窗口内合理组织检索结果。
- **LLM 能力依赖**：最终回答质量仍取决于模型的理解与推理能力。

### Q5: RAG 实现的基本步骤
1. 读取本地文档并切分为语义块（Chunk）。
2. 使用 Embedding 模型将各块向量化。
3. 存储向量至向量数据库（如 ChromaDB）。
4. 用户提问时进行向量相似度检索，召回 Top-K 相关片段。
5. 将召回片段与用户问题组合成最终 Prompt，交由 LLM 生成回答。

## 三、Embedding 模型选型与评估

### 评估 Embedding 模型的八大关键要素

| 要素 | 说明 |
|------|------|
| **上下文窗口** | 模型单次能处理的最大 Token 数。长文档（>512 tokens）建议选用 8192 tokens 及以上模型（如 BGE-M3）。 |
| **分词方式** | 专业领域（医学、法律）应优先选择子词分词（如 BPE），确保术语完整性。 |
| **向量维度** | 通用场景 768-1536 维；高精度需求可选用 2000 维以上。 |
| **检索质量** | RAG 效果的核心决定因素，需关注权威评测榜单（如 C-MTEB）。 |
| **模型大小与资源占用** | BGE-M3 显存约 6.8GB，M3E 内存仅 3.2GB，需根据硬件条件权衡。 |
| **推理速度与延迟** | M3E 可达 2300 QPS，实时应用需优先考虑低延迟模型。 |
| **成本** | 开源模型需承担算力成本，商业 API 按 Token 计费，大规模使用时差异明显。 |
| **领域适应性** | 通用模型在垂直领域表现可能不足，可考虑专用模型（如 BioBERT、LegalBERT）。 |

### 主流开源模型对比：BGE vs M3E

#### BGE 系列（智源研究院 BAAI）
- **定位**：中文领域的“通用嵌入模型”，旗舰型号 **BGE-M3** 具备多功能、多语言、多粒度特性。
- **核心能力**：
  - 支持稠密、稀疏、多向量三种检索模式。
  - 支持 100+ 语言，上下文窗口 8192 Token。
  - 预训练使用了海量中英文样本对（中文 1.2 亿对，英文 2.32 亿对）。
- **适用场景**：多语言、长文档、高精度要求的通用 RAG 系统。

#### M3E 系列（MokaAI）
- **定位**：专注中文的轻量级嵌入模型。
- **核心能力**：
  - 使用 2200 万+ 中文句对训练，中文语义把握精准。
  - 模型体积约为 BGE-M3 的 60%，内存占用 3.2GB，推理速度 2300 QPS。
- **适用场景**：资源受限环境、纯中文短文本、追求高性价比的 RAG 应用。

### 选型决策速查表

| 场景 | 推荐模型 | 理由 |
|------|----------|------|
| 通用中文 RAG（不确定选哪个） | **BGE-M3** | 综合性能最强，中文榜单领先。 |
| 纯中文短文档（<512 tokens） | BGE-large-zh | 针对短文本精度优化。 |
| 硬件资源有限 / 追求低成本 | **M3E** | 轻量、高速、低内存占用。 |
| 超长文档分块（如整章法律条文） | BGE-M3 或 Jina Embeddings v2 | 支持 8192 长上下文。 |
| 已深度绑定 OpenAI 生态 | text-embedding-3-small/large | 保持技术栈一致性。 |

## 四、ReRank 重排序模型

### Q6: 什么是 ReRank 模型？
ReRank 模型是一种对 RAG 第一阶段检索结果进行**二次排序**的模型。

**工作流程**：
1. RAG 第一阶段使用 Embedding 模型从向量库中召回 Top-K 个候选片段（如 K=20）。
2. 将用户问题与这 20 个片段逐一配对，送入 ReRank 模型计算**更精细的相关性分数**。
3. 根据新分数重新排序，选出最相关的 Top-N（如 N=3）送入 LLM。

### 为什么需要 ReRank？
- **弥补 Embedding 的粗糙度**：Embedding 模型追求速度，在语义相似度计算上可能存在偏差，ReRank 模型可进行更精确的语义匹配。
- **提升上下文纯度**：在 LLM 上下文窗口有限的情况下，通过重排序确保输入的内容与问题最相关。

### 主流 ReRank 模型推荐
- **BGE-Reranker-v2-m3**：与 BGE-M3 同系列，支持多语言，重排序效果 SOTA。
- **Cohere Rerank**：商业 API，效果好但需付费。
- **Cross-Encoder 模型**：如 `cross-encoder/ms-marco-MiniLM-L-6-v2`，适合英文场景。

### ReRank 与 Embedding 模型的区别

| 对比维度 | Embedding 模型 | ReRank 模型 |
|----------|---------------|-------------|
| **任务目标** | 将文本转换为向量，用于快速相似度检索 | 直接计算问句与文档片段的精确相关性分数 |
| **计算方式** | 对文档和查询分别编码，通过向量点积/余弦相似度比较 | 将查询和文档拼接后联合编码，输出单一分数 |
| **速度** | 极快，适合海量召回 | 较慢，仅用于 Top-K 结果精排 |
| **应用阶段** | RAG 第一阶段（召回） | RAG 第二阶段（重排） |

## 五、面试常见追问补充

### Q7: 文档切片时，Chunk Size 和 Overlap 如何选择？
- **Chunk Size**：取决于文档类型和 Embedding 模型的上下文窗口。说明书类短文可选 256-512；技术文档可选 512-1024。需确保每个 Chunk 包含完整的语义单元。
- **Overlap**：一般设为 Chunk Size 的 10%-20%，防止关键信息被切在边界导致检索遗漏。

### Q8: 向量数据库选型（ChromaDB vs FAISS）
- **ChromaDB**：嵌入式数据库，开箱即用，自带元数据过滤和持久化，适合原型开发和中小规模应用。
- **FAISS**：Facebook 开源的底层向量检索库，性能极高，但需自行处理持久化和分布式部署，适合大规模生产环境。

### Q9: 如何评估 RAG 系统的效果？
- **检索阶段**：使用 Hit Rate、MRR（平均倒数排名）评估召回是否包含正确答案。
- **生成阶段**：使用 RAGAS 框架评估答案忠实度（Faithfulness）和答案相关性（Answer Relevancy）。

---

### 为什么要评估？从“感觉”到“量化”

RAG是一个包含检索、排序、生成等多个环节的复杂系统，任何一个环节出问题都可能影响最终效果。只凭“感觉”来优化，不仅低效，而且不准确。评估的核心目的，就是为了回答三个关键问题：
1.  **定位问题**：是检索没找到信息，还是生成时曲解了信息？
2.  **量化效果**：当前的系统在准确性、相关性等方面到底能得多少分？
3.  **指导优化**：根据得分，我们应该优先优化哪个部分？

### 如何评估？关键指标拆解

#### 阶段一：检索评估 (Retrieval Evaluation)
这个阶段的核心是评估系统能否从知识库中**精准、全面**地找到与用户问题相关的信息。

*   **召回率 (Recall)**: 衡量检索系统是否找得**全**。
    *   **公式**：`召回率 = 检索到的相关文档数 / 知识库中所有相关文档总数`
    *   **解读**：高召回率确保系统没有遗漏关键信息。在合规、风控等对信息完整性要求高的场景，高召回率至关重要。

*   **精确率 (Precision)**: 衡量检索系统是否找得**准**。
    *   **公式**：`精确率 = 检索到的相关文档数 / 检索到的所有文档总数`
    *   **解读**：高精确率意味着检索到的“噪音”少，能帮助LLM更聚焦。

*   **平均倒数排名 (MRR)**: 衡量系统定位**第一个相关文档**的**速度**。
    *   **公式**：`MRR = 1 / 第一个相关文档的排名`
    *   **解读**：MRR越高，说明相关结果越靠前。对于用户只关心第一个答案的场景（如问答机器人），该指标非常重要。

*   **平均精度均值 (MAP)**: 综合考虑所有相关文档的**排名和数量**。MAP越高，说明系统不仅找得全，而且排得准。

#### 阶段二：生成评估 (Generation Evaluation)
这个阶段的核心是评估LLM生成的答案是否**准确、相关、忠于事实**。

*   **忠实度 (Faithfulness)**: 评估生成的答案是否**严格基于**检索到的文档。
    *   **意义**：这是衡量“幻觉”程度的核心指标。高忠实度意味着答案的每一句话都能在参考资料中找到依据。

*   **答案相关性 (Answer Relevancy)**: 评估生成的答案是否**直接回应**了用户的问题。
    *   **意义**：防止LLM答非所问。如果答案相关但检索内容不相关，可能LLM在依靠自己的“记忆”作答，这是危险的。

*   **上下文相关性 (Context Relevancy)**: 评估LLM在生成答案时，是否**真正使用**了检索到的信息。
    *   **意义**：它可以暴露出检索的内容是否被有效利用，防止出现“检索到了但没用上”的浪费。

*   **答案正确性 (Answer Correctness)**: 从语义层面判断生成答案与标准答案的**相似度**。通常需要标准答案作为参考。

*   **抗噪能力 (Noise Sensitivity)**: 评估当检索到的上下文中混入无关信息时，LLM是否能**不受干扰**，依然给出正确答案。

### 用什么工具？主流评估框架

| 框架名称 | 核心特点 | 适用场景 |
| :--- | :--- | :--- |
| **RAGAS** | **最全面、最流行**的RAG评估框架。提供从检索到生成的多种无参考（reference-free）指标，并可自动生成测试集。 | 适合希望**全面量化**RAG系统性能，进行**指标驱动开发（MDD）** 的团队。 |
| **DeepEval** | 提供超过**40种开箱即用**的评估指标，非常适合需要**企业级全链路监控**的场景。 | 适合希望**快速搭建**起全面评估体系，或需要深度集成到CI/CD流程中的团队。 |

| **OpenAI Evals** | OpenAI官方提供的轻量级评估工具，尤其适合**基于OpenAI接口**构建的简单RAG测试。 | 适合**快速验证**想法，或进行**单元测试**级别的评估。 |

### 如何构建评估流程

评估本身不是目的，通过评估来迭代优化系统才是。建议遵循“指标驱动开发（MDD）”的理念，将评估融入到开发的全流程。

1.  **构建高质量评估数据集**
    *   **来源**：可以从真实用户日志中提取，也可以使用框架（如RAGAS）的自动测试集生成功能，从文档中自动创建问答对。
    *   **组成**：确保数据集覆盖了**常见问题、边缘案例、对抗性问题**等不同维度。

2.  **分层评估与持续监控**
    *   **离线评估**：在开发阶段，使用评估数据集进行全面的离线测试，确保每次代码或配置变更都不会引入性能回退。
    *   **在线监控**：在系统上线后，建立对关键指标（如召回率、忠实度）的实时或准实时监控。一个推荐的分层监控策略是：
        *   **基础层（天级）**：运行自动化指标测试。
        *   **监控层（实时）**：捕获用户的负反馈（如点踩、投诉）。
        *   **审计层（周级）**：对线上结果进行抽样深度分析。

3.  **设计闭环的评估-优化实验流程**
    *   **发现问题**：通过指标面板，比如发现“忠实度”得分下降。
    *   **定位瓶颈**：进一步分析是哪个组件出问题，比如“上下文相关性”得分正常，但“忠实度”低，说明问题可能出在生成阶段。
    *   **实施优化**：针对性地进行调整，如优化Prompt或更换模型。
    *   **A/B测试**：对优化后的系统和新系统同时进行评估，用数据证明改进是否有效。
    *   **部署与监控**：将有效的优化部署上线，并纳入监控体系。

通过建立这样一套“评估-监控-优化”的闭环，可以持续不断地提升RAG系统的质量和可靠性。
