## 一、项目概览：RAGFlow 是什么

RAGFlow 是一个开源 **RAG（检索增强生成）引擎**，它解决的核心问题是：

> 用户有一堆私有文档（PDF、Word、Excel...），如何让大模型能“读懂”这些文档并基于它们回答问题？

这个项目将整个流程拆成几个关键环节：
1. **文档解析** — 把 PDF/Word/Excel 变成纯文本
2. **切片（chunk）** — 把长文本切成小块
3. **向量化（embedding）** — 把每块文本转成数学向量
4. **检索生成** — 用户提问时，找到最相关的块，拼进提示词里喂给大模型

我选择从**第 2 步“切片”**切入，因为它是承上启下的核心环节。

### 阶段 1：定位核心文件

**问题**：项目有几十个文件夹，从哪看起？

**做法**：
1. 用 PyCharm 打开项目，按两下 `Shift` 打开全局搜索
2. 搜索 `chunk`，在 `rag/app/` 目录下找到 `naive.py`
3. 打开文件，用 `Ctrl+F12` 查看文件结构，找到 `chunk()` 函数（约 200 行）

**发现**：`chunk()` 是一个大分支结构，根据文件后缀名（`.pdf` / `.docx` / `.xlsx` / `.md` ...）走不同逻辑。  
→ **这让我第一次看到“策略模式”在真实项目中的样子**：每种文件类型有独立的解析器，但最后都汇合到 `naive_merge()` 进行切片。
- **策略模式**：把一组可以互相替换的算法分别封装好，然后根据需要随时切换，而不改动调用它们的代码。**上层解析可替换，下层切片逻辑不变**

### 阶段 2：深入 `chunk()`，理解 PDF 处理分支

**具体代码段**（`naive.py` PDF 分支）：

```python
elif re.search(r"\.pdf$", filename, re.IGNORECASE):
    # 选一个“阅读器”：DeepDOC / MinerU / PlainText
    parser = PARSERS.get(name, by_plaintext)
    sections, tables, pdf_parser = parser(
        filename=binary,
        from_page=from_page,
        to_page=to_page,
        ...
    )
    # 把段落按 token 数合并成块
    chunks = naive_merge(sections, chunk_token_num, delimiter, overlapped_percent)
```

**学到的知识点**：
- `PARSERS` 是一个字典，Key 是 `"deepdoc"` / `"mineru"` / `"plaintext"`，Value 是对应的解析器类。这是**工厂模式 + 策略模式**的工程实践。
- 不论用哪种解析器，输出都是 `sections`（文本段落列表）和 `tables`（表格列表），**接口统一，上层可任意替换实现**。

### 阶段 3：`naive_merge`——切片核心算法

**为什么盯这个函数？**  
因为所有文件类型（PDF、Word、TXT...）解析完后，都会调用 `naive_merge()` 来真正切成块。它是整个模块的**算法核心**。

**函数签名**：
```python
def naive_merge(sections, chunk_token_num=128, delimiter="\n。；！？", overlapped_percent=0):
```

**内部核心数据结构**（我读代码时发现的）：
```python
cks = [""]       # 存储每个块的内容字符串
tk_nums = [0]    # 存储每个块当前的 token 数，和 cks 一一对应
```

**学到的底层逻辑**：
- 使用**两个同步增长的数组**，`cks[-1]` 永远是“正在填充的当前块”，`tk_nums[-1]` 是它当前的 token 数。
- 这样设计比用一个单独的 `current_chunk` 变量更灵活——可以同时维护多个块，也方便以后并行处理。

---

### 阶段 4：判断条件——块什么时候“满”？

**在项目中找到的关键代码行**：
```python
if cks[-1] == "" or tk_nums[-1] > chunk_token_num * (100 - overlapped_percent) / 100:
```

**拆解这个条件**：
- `cks[-1] == ""`：当前块是空的（刚新建），直接往里放。
- `tk_nums[-1] > chunk_token_num * (100 - overlapped_percent) / 100`：
  - 假设 `chunk_token_num=512`, `overlapped_percent=10`
  - 则警戒线 = `512 * (100-10) / 100 = 460.8`
  - **当前块的 token 数超过 460 时，就触发“新建块”**

**为什么是 460 而不是 512？**  
→ 预留了 10% 的空间给**重叠内容**。这样下一块的开头可以复制上一块结尾的部分内容，避免语义在边界被割裂。

### 阶段 6：写简化版，验证猜想

**代码**（`test_minimal.py`）：
```python
def naive_merge_simple(sections, chunk_token_num=128):
    tk_nums = [0]
    cks = [""]
    for text, _ in sections:
        new_tokens = len(text.split())   # 用单词数模拟 token 数
        if cks[-1] == "" or tk_nums[-1] + new_tokens > chunk_token_num:
            cks.append(text)
            tk_nums.append(new_tokens)
            print(f"新建块，token数: {new_tokens}")
        else:
            cks[-1] += "\n" + text
            tk_nums[-1] += new_tokens
            print(f"追加到块，当前token数: {tk_nums[-1]}")
    return [c for c in cks if c]

if __name__ == "__main__":
    fake = [
        ("Hello world. " * 40, ""),   # 约 80 词
        ("Goodbye. " * 40, ""),       # 约 40 词
    ]
    result = naive_merge_simple(fake, chunk_token_num=100)
    print("最终块数:", len(result))
```

**运行结果**：
```
新建块，token数: 80
新建块，token数: 40
最终块数: 2
```

**为什么这个输出证明我“学会了”？**
- 第一段 80 token，放入空块 → 触发“新建块”
- 第二段 40 token，但 `80 + 40 = 120 > 100` 上限 → 又触发“新建块”
- 最终两个块，说明算法确实在“**装满一块就换下一块**”

→ 我的简化版行为和真实 `naive_merge` 完全一致，只是项目里用 `rag_tokenizer.tokenize()` 而我用了 `.split()` 来模拟 token 计数。

### 阶段 7：对比简化版与项目源码，融会贯通

| 简化版 | 真实项目 | 为什么不同 |
|--------|----------|-----------|
| `len(text.split())` | `rag_tokenizer.tokenize(text)` | 真实项目要兼容中英文，必须用分词器 |
| `tk_nums[-1] + new_tokens > chunk_token_num` | `tk_nums[-1] > chunk_token_num * (100-overlapped)/100` | 真实项目有重叠机制，预留了空间 |
| `delimiter = "\n"` | `delimiter = "\n!?。；！？"` | 真实项目可按中文标点分段 |
| 模拟数据 | 真实 PDF 解析出的 `sections` | 真实数据来自 OCR/文档解析器 |

## 三、学到的可迁移底层逻辑

1. **“贪心合并”是切片最通用的策略**：遍历 → 累加 → 超限就新建，时间复杂度 O(n)，简单但有效。

2. **Token 感知是 RAG 的刚需**：大模型按 token 计费和限制窗口，所以切片必须用 token 数而不是字符数做判断。

3. **预留重叠空间是工程优化**：`overlapped_percent` 不是学术概念，而是为了在边界处保留语义连贯性，同时提升检索召回率。

4. **策略模式让系统可扩展**：`PARSERS` 字典让解析器可插拔，这是“开闭原则”（对扩展开放，对修改关闭）的典型实践。

5. **简化实验是理解复杂项目的利器**：当依赖链爆炸时，剥离外壳、只留核心算法，用 print 观察行为——这种方法可以迁移到任何开源项目的学习中。
